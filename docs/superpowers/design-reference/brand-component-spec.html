Brand Component Guideline

LexSeek 品牌色组件规范

本稿用于确认全站品牌主题的视觉边界：品牌色负责主行动、焦点、导航选中和重点展示；成功、警告、错误等业务状态继续使用语义颜色。后续落地时，主题切换只保留亮色和暗色。

主要色 用于页面主行动、链接、焦点环、当前导航项、分页选中态。不要用于错误或警告。
默认色 用于普通按钮、背景、边框、未激活标签、次要状态。默认色要保持安静，不抢主行动。
警告色 用于风险提示、预扣、即将过期、余额不足等“需要处理但不是失败”的状态。
错误色 用于表单错误、支付失败、删除、撤销、不可恢复操作。错误色不使用品牌渐变。
成功色 用于完成、可用、已生效、支付成功。不要把成功按钮做成品牌蓝，以免用户误解为普通主操作。

字体、圆角与布局

字体层级

页面标题 24-30px，卡片标题 16-18px，正文 14-15px，辅助说明 12-13px。业务后台不使用营销页超大标题。

圆角

按钮和输入框 8px，卡片 12-14px，弹窗 14-16px。避免过度圆润，保持法律工具的专业感。

间距

表单项 12-16px，卡片内边距 16-24px，页面边距桌面 24-32px，移动端 16px。

按钮规范

每个操作区最多一个品牌主按钮。取消、关闭、刷新、复制、详情、重置等次要动作使用 outline 或 ghost。

主按钮 提交、支付、购买、开始审查、创建案件、执行检索。使用品牌按钮渐变，白字。
次按钮 取消、关闭、返回、复制、下载、刷新、查看详情。使用 outline 或 ghost，不使用品牌渐变。
危险按钮 删除、作废、取消订单等可能造成损失的操作。使用红色，并必须配确认弹窗。
加载态 按钮文字保留用户动作，如“创建中...”“支付中...”，左侧显示 loading 图标，按钮不可重复点击。
移动端 底部或表单尾部主按钮可以全宽；同一行多个按钮时，主按钮优先在左或右需按页面上下文统一。

提示、状态与徽章

提示组件用于告诉用户“发生了什么、是否需要处理、下一步是什么”。错误提示必须明确可恢复动作。

默认提示

用于普通说明、空状态补充、流程解释。

主要提示

用于品牌相关引导、当前步骤说明、可继续操作的信息。

警告提示

积分即将不足，继续操作可能产生预扣。

错误提示

支付失败，请检查网络后重试，或联系客服处理。

成功提示

订单已支付成功，权益将在数秒内到账。

现行有效 已完成 预扣 异常 已关闭
  • 徽章只承载状态，不承载长句；长说明放在正文或 Tooltip。
  • 业务状态色不跟随品牌色变化，避免“蓝色既是品牌又是失败/成功”的误读。
  • 同一列表内的状态命名要稳定，例如“有效 / 已结算 / 已作废”，不要同义词混用。

表单组件

表单优先清晰和可恢复。焦点环使用品牌主色，错误使用红色，必填星号使用错误色。

用于登录、注册、找回密码。
手机号格式不正确
输入框 默认白底、细边框；聚焦时品牌色边框和 3px 柔和焦点环。
勾选框 选中态使用品牌按钮渐变 bg-gradient-brand-button，未选中保持中性边框；不要退回单色 primary。
校验错误 红色边框、红色说明。错误说明要说明如何修正，不只写“错误”。
选择器 短选项使用 Select；长列表和可搜索项使用 Combobox / Popover。
开关 启用态使用品牌按钮渐变 bg-gradient-brand-button，关闭态使用 muted/input；焦点环使用品牌色，不使用单色主题色。
禁用态 降低透明度，不改变布局，不隐藏字段。需要告诉用户为什么不可用。

卡片规范

卡片用于组织信息，不做过重装饰。可点击卡片 hover 时轻微上浮和品牌边框即可。

AI 对话：现有结构锁定 + 配色规范

结构以通用问答当前实现为准：页面外层、会话列表、会话顶栏、AssistantChatPanel、AiChat、虚拟消息列表、工具卡、队列条、输入框头部/正文/底部工具栏都不能被简化重画。

页面外层

保留 relative isolate flex flex-1 min-h-0 gap-4 overflow-hidden p-3 sm:p-4、移动端会话抽屉遮罩和桌面停靠侧栏。配色只调整遮罩、页面底、卡片边框。

AssistantSessionList

保留“新会话按钮、搜索框、loading 骨架、空状态、会话列表、选中态左侧品牌色条”。配色只规定新会话按钮渐变、搜索框 muted 底、选中项品牌渐隐底纹和普通项 hover。

会话顶栏

保留移动端菜单、小索头像、标题/创建时间、标题内联编辑、编辑和删除图标按钮。配色只规定头像品牌外圈、确认用 primary、取消用 muted、删除用 destructive。

AssistantChatPanel / AiChat

保留队列暂停横幅、AiChatQueueChips、InterruptDispatcher、材料选择器、单/双面板能力、ClientOnly 和虚拟消息列表。配色只规定队列 amber、工具卡 card/border、加载 primary、错误 destructive。

消息列表

保留 Conversation、ConversationEmptyState、AiMessageListVirtual、Message/MessageContent、Reasoning、MessageResponse、AiToolRenderer。配色只规定用户消息、助手正文、推理块、附件气泡和工具结果卡的明暗适配。

AiPromptInput

保留文件 chip 头部、拖拽覆盖层、textarea、底部工具栏、上传按钮、深度思考开关、发送/停止/加入队列按钮。配色只规定 focus ring、文件状态徽章、思考开关、发送按钮、停止按钮和队列角标。

  • 主动作“新会话”“开始新对话”“发送”使用品牌渐变；“停止”使用 destructive；“加入队列”按现有 Button 结构，可用 primary，但不能改结构。
  • 会话选中保持左侧品牌渐变色条和横向渐隐底纹；普通会话 hover 使用 primary/8，不把每条会话都做成强蓝块。
  • 输入框PromptInput 外框使用 border-primary/35、shadow-primary/15，拖拽态使用 bg-primary/10 border-primary。
  • 消息与工具保留 MessageContent 的用户/助手分支；附件消息继续透明承载附件卡，工具卡统一 card + border，不用大面积品牌渐变。
  • 队列与中断排队、暂停、余额、失败等业务状态按 amber / red / green 语义色，不用品牌蓝替代。

全局工作台框架：侧边栏、顶栏、视图切换

结构以 dashboardLayout、navMain、ThemeToggle、ViewModeToggle 为准。这里补的是全站页面骨架配色，避免只改内容区而漏掉导航态。

  • 侧边栏背景工作台根容器保持 theme-brand；侧边栏使用 var(--wash-sidebar)，不要在菜单项内写死白底或深色背景。
  • 菜单选中态保留左侧 3px 品牌竖条和青蓝渐隐底纹；文字使用 text-primary，普通项 hover 用 bg-primary/[0.08]。
  • 顶栏使用 bg-background/70 backdrop-blur-md border-b，主题切换只呈现亮色/暗色，不再出现多色主题入口。
  • 视图切换保留图标按钮结构；激活项为 bg-background shadow-sm text-primary，未激活项为 muted 文字。

案件详情：侧栏、底部 Tab、分析模块、小索浮窗

暗色模式完整规范

本节用于直接交给 AI 执行页面重构：暗色模式只改变明暗层级，不改变品牌主题。全站保留 theme-brand 品牌主题和 dark 明暗切换，不再保留多色主题入口。

一、暗色模式的产品原则

暗色不是“黑色版本”，而是低亮度工作台。用户需要长时间阅读法律文本、表格和 AI 输出，所以暗色要降低眩光，同时保留足够的边界、层级和状态识别。

必须保持一致

  • 页面根部仍使用品牌主题容器，暗色只通过 html.dark 触发。
  • 主操作仍使用品牌按钮渐变，不能改成灰色、绿色或其它主题色。
  • 链接、焦点、选中、当前导航使用暗色抬亮后的品牌蓝。
  • 成功、警告、错误继续使用语义色，不被品牌蓝替代。
  • 卡片、表格、弹窗、侧边栏都必须有可见边界，不能只靠阴影分层。

必须避免

  • 不要写死 bg-white、text-zinc-900、border-zinc-200 作为暗色页面主体。
  • 不要使用 theme-blue、theme-green、theme-rose 等多色主题类。
  • 不要在暗色下扩大阴影来表达悬浮，优先使用边框、背景透明度和轻微亮度变化。
  • 不要让红、黄、绿状态色饱和成荧光块，使用低透明背景加亮色文字。
  • 不要让多个强品牌按钮同时出现在一个操作区。

二、暗色 token 对照

AI 重构页面时优先使用语义 token，而不是直接写颜色。下面的色值是视觉目标，项目实现应落到全局 token 和 Tailwind 语义类上。

页面背景 bg-background

用于页面根背景。品牌工作台可叠加 var(--wash-page)，但不能直接用纯黑。

卡片/弹窗 bg-card

用于卡片、弹窗、表格单元格。和页面背景之间必须有边框或背景差。

弱背景 bg-muted

用于表头、hover、输入框辅助底、AI 助手消息。不要用于主按钮。

边框/分割线 border-border

暗色边框必须可见但不刺眼，常用 rgba(255,255,255,.10-.14)。

品牌主色 text-primary

用于链接、焦点、当前项、重点数字。暗色下要比亮色更亮。

品牌辅助强调 tint / strong

用于深蓝品牌延展、图标块前景、少量辅助强调，不用于错误或警告。

七、AI 重构检查清单

  • 先识别页面角色

    判断页面是工作台、列表、详情、表单、支付、AI 对话还是文档阅读；再套用对应结构，不要把所有页面做成同一种卡片堆叠。

  • 结构以项目现有组件为准

    本 HTML 的静态示意只表达配色和状态规则，不是组件结构蓝图；重构时不得把现有复杂组件替换成文档里的简化 div 结构。

  • 替换硬编码颜色

    把主体区域的白底、黑字、灰边替换成 background/card/foreground/muted/border/input 等语义 token。

  • 保留业务语义色

    成功用绿，警告和待支付用 amber，错误和删除用 red/destructive，品牌蓝只用于品牌和主行动。

  • 控制主按钮数量

    一个操作区域只能有一个品牌渐变按钮；其它按钮必须降级为 outline、ghost 或图标按钮。

  • 检查可读性和边界

    暗色下正文、表格、输入框、弹窗、下拉菜单、toast、分页、选中态都必须有足够对比和明确边界。

  • 同时验收亮色和暗色

    任何页面重构完成后，都要检查亮色不丢品牌感，暗色不残留白底、黑字、低对比文字或不可见边框。

AI 执行约束

本章节是给 AI 的硬性验收协议。执行品牌主题迁移时，不能只看静态示意图，必须先对照项目现有实现，再用浏览器验证真实样式。

一、任务边界

  • 只改配色允许调整品牌色、语义状态色、焦点、hover、active、selected、disabled、链接和主按钮；不允许改页面信息架构。
  • 结构不重画不得新增顶部渐变面板、筛选卡片外框、表格阴影、移动卡片重排、详情区卡片包裹、营销式 hero 或额外统计信息。
  • 以现有组件为准先阅读目标页面和子组件，保留现有布局、按钮位置、表格列、移动端结构、抽屉/弹窗结构和业务文案。
  • 只在必要处补作用域页面根节点、Sheet、Dialog、Dropdown、SelectContent 等可能脱离父级作用域的区域，需要确认品牌 token 是否实际生效。

二、禁止变更清单

后台标题区 保持原有紧凑标题 + 说明 + 右侧动作，不加渐变背景、不加统计块、不增加高度。
筛选表单 保持原有横向裸排，不额外包卡片、不加标题说明、不改变控件顺序。
表格 保持原有边框和表头样式，不加 shadow，不把表格改成卡片堆叠；只改必要的状态色和交互色。
移动卡片 保持现有信息顺序和密度，不新增 hover 阴影或拆分金额/标题结构，除非原组件已经如此设计。
详情抽屉 保持现有 section 和间距，不把每段内容重新包成带阴影卡片。
业务状态 成功、警告、错误、取消、过期等状态使用语义色，不随品牌蓝或当前主题色漂移。

三、焦点态验收

  • 不要只看 class

    表单控件可能同时带有 focus-visible:border-ring、focus-visible:ring-ring/50 和品牌覆盖类；必须读取浏览器最终计算样式。

  • 品牌焦点标准

    输入框、日期框、下拉触发器、Textarea 聚焦后，边框应为品牌蓝 rgb(30, 158, 237)，光晕应为 rgba(30, 158, 237, 0.36)，不能显示 green / rose / orange / violet 等旧主题色。

  • 必须用非品牌主题反测

    验收时先切到 green、rose 或其它旧主题色，再刷新目标页面并聚焦表单；如果焦点仍跟随旧主题色，说明迁移失败。

  • Portal 组件单独验收

    Select 下拉、Sheet、Dialog、DropdownMenu 可能渲染在页面根节点之外，不能假设父级 theme-brand 一定覆盖到它们。

四、浏览器验收步骤

1. 打开真实页面 使用已登录浏览器打开目标路由，不用静态 HTML 推断效果。
2. 聚焦关键控件 依次点击输入框、日期框、下拉框、Textarea、主按钮、分页当前项。
3. 读取计算样式 用 getComputedStyle 检查 borderColor、boxShadow、--tw-ring-color、文字色和背景色。
4. 对照禁止清单 确认没有新增渐变面板、筛选卡片、表格阴影、卡片嵌套、额外文案或结构重排。
5. 亮暗色双测 同一页面必须检查亮色和暗色；暗色不能残留白底黑字，亮色不能丢失品牌焦点。

五、失败示例

  • 把后台标题区改成大面积品牌渐变面板。
  • 给原本裸排的筛选表单新增外框、标题或说明。
  • 给原本无阴影的表格新增阴影。
  • 只在页面根节点加 theme-brand，但没有验证 Select、Sheet、Dialog 里的控件是否生效。
  • 只检查 DOM class，不检查最终 borderColor 和 boxShadow。
  • 在默认主题下看起来正确，但切到 green / rose / violet 后焦点色仍跟随旧主题。

全站落地规则

应该做

  • 全站默认使用品牌主题，只保留亮色和暗色切换。
  • 主操作使用品牌渐变，次要操作使用 outline / ghost。
  • 焦点、链接、导航当前项使用品牌主色。
  • 成功、警告、错误保持语义颜色，不跟随品牌色。
  • 所有图标按钮使用可识别图标，并提供 hover / focus 状态。
  • 弹窗必须有标题和说明，危险操作必须说明后果。

不应该做

  • 同一操作区放多个品牌渐变按钮。
  • 把取消、关闭、刷新、复制做成主按钮。
  • 用品牌蓝表示错误或失败。
  • 在业务系统里使用过大的营销页标题。
  • 使用 emoji 图标；项目内应统一使用 lucide 图标。
  • 用原生 confirm / alert 处理业务确认。
  • 把本 HTML 的静态示意结构直接替换项目现有组件结构。

组件落地映射

主按钮 bg-gradient-brand-button text-white，只用于页面最关键动作。
次按钮 variant="outline" 或 variant="ghost"，用于取消、详情、刷新、复制。
品牌强调 text-primary、bg-primary/10、border-primary/20。
图标底 var(--tint-sky-bg)、var(--tint-mint-bg)、var(--tint-navy-bg)、var(--tint-amber-bg)。
卡片 rounded-xl border bg-card，hover 使用品牌边框和轻阴影。
错误 text-destructive 或红色语义 token，不使用品牌渐变。
警告 使用 amber 语义色，适合余额、过期、预扣、风险提示。
深浅色 组件必须同时检查亮色和暗色，尤其是渐变按钮白字、淡彩底和红/黄状态。

确认后建议把本规范拆成全局 token、按钮规范、业务状态规范、页面巡检清单四个执行项，再逐模块落地。
