# 法律向量重建脚本并行化设计

## 概述

将 `server/scripts/rebuildLawEmbeddings.ts` 从串行处理改为并行处理，使用 `p-limit` 库实现 concurrency=5 的滑动窗口，并增加整批失败重试机制。

## 背景

当前脚本逐条串行处理法律条文的 embedding 重建（删除旧嵌入 → 重建新嵌入），由于 embedding API 调用是 I/O 密集型操作，串行处理效率极低，跑了几天仍未完成。

## 设计

### 依赖

新增 `p-limit` 依赖，用于控制 Promise 并发数。

### 核心变更

1. **移除 `BATCH_SIZE` 分批逻辑**：不再按固定批次切片遍历
2. **新增 `CONCURRENCY = 5` 常量**：控制最大并发数
3. **抽取 `processLegal()` 函数**：封装单条法律的删除+重建逻辑
4. **两轮执行**：
   - 第一轮：全量法律并发提交到 `p-limit(5)` 滑动窗口
   - 第二轮：对第一轮失败项使用相同并发度重试一次
5. **进度跟踪**：每条完成时输出当前进度

### 数据流

```
查询全部法律
  → 全量提交到 p-limit(5) 并发队列
  → 每条独立 try/catch，成功计数 / 失败收集
  → 第一轮完成
  → 失败项不为空？重试一轮（同 p-limit(5)）
  → 最终汇总输出
```

### 错误处理

- 单条法律处理失败：捕获异常，记录 `{ id, name, error }`，不影响其他并发任务
- 重试轮：对第一轮失败项再跑一次，仍失败的记录最终失败列表
- 脚本 exit code：有最终失败项返回 1，全部成功返回 0

### 进度日志

```
[12/500] 处理: 中华人民共和国民法典 (xxx-id)
  删除旧嵌入 42 条
  重建完成
[13/500] 处理: 中华人民共和国刑法 (yyy-id)
  重建失败: Error message
当前进度: 成功 10, 失败 2, 共 12/500
```

### 接口不变

- `--start-from=<legalId>` 和 `--limit=N` 参数保持不变
- 全局依赖初始化逻辑不变
- 最终失败 ID 列表输出格式不变

## 影响范围

仅修改 `server/scripts/rebuildLawEmbeddings.ts` 一个文件，新增 `p-limit` 依赖。
